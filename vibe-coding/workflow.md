# Vibe Coding 完整工作流

> Vibe Coding：通过自然语言描述需求，让 AI 完成代码编写，你负责理解、测试和迭代。

---

## 什么时候适合 Vibe Coding

**适合：**
- 复试项目（时间紧，快速出效果）
- 验证一个想法是否可行
- 不熟悉的技术栈的快速原型

**不适合：**
- 你完全不需要理解代码的场景（面试会穿帮）
- 超大型复杂系统（上下文长度限制）

---

## 完整工作流

### 阶段一：项目初始化

**目标：** 让 AI 生成项目骨架，你确认结构合理后再开始。

**使用** → [prompt-templates.md](prompt-templates.md) 中的「VC-01 项目初始化 prompt」

**验收标准：**
- 目录结构清晰，你能说出每个文件夹的作用
- 能成功运行（哪怕只是 Hello World）
- 主要依赖已安装

**常见问题：**
- AI 生成的项目结构过复杂 → 让它简化，说"只保留核心功能，去掉暂时用不到的模块"
- 依赖安装失败 → 把完整错误信息给 AI，让它解决

---

### 阶段二：功能开发循环

每个功能按以下循环执行：

```
1. 用自然语言描述这个功能要做什么
2. AI 实现代码
3. 你阅读代码，不懂的部分让 AI 解释
4. 运行测试，确认功能正常
5. git commit（每个功能一个提交）
6. 进入下一个功能
```

**使用** → [prompt-templates.md](prompt-templates.md) 中的「VC-02 功能需求描述 prompt」

**关键原则：**
- **每个功能做完就 commit**：出错可以回滚
- **读懂再继续**：不理解上一步的代码，不要开始下一步
- **一次只做一件事**：不要让 AI 一次性实现多个功能

---

### 阶段三：调试修复

遇到 bug 时，不要只把错误信息扔给 AI。

**有效的 bug 描述格式（4要素）：**
1. 我期望的行为是什么
2. 实际发生了什么
3. 错误信息（完整复制，不要截图）
4. 我已经尝试过什么

**使用** → [prompt-templates.md](prompt-templates.md) 中的「VC-03 Bug 修复 prompt」

**常见陷阱：**
- AI 修复了一个 bug，引入了另一个 → 每次修复后都要测试完整功能，不只测刚修的那里
- AI 反复无法解决同一个 bug → 说明上下文已经混乱，开新对话，重新描述问题

---

### 阶段四：代码理解（面试准备关键）

项目完成后，逐个模块过一遍，确保你能讲清楚每一行关键代码。

**使用** → [prompt-templates.md](prompt-templates.md) 中的「VC-04 代码解释 prompt」

**验收标准（对自己的提问）：**
- 这个模块/函数的作用是什么？（能用一句话说）
- 它用了什么核心逻辑？（不需要记每行，但要知道算法思路）
- 如果面试官问"这里为什么这样写"，我能回答吗？

---

### 阶段五：代码优化（可选）

如果时间允许，可以让 AI 帮你优化代码质量。

**使用** → [prompt-templates.md](prompt-templates.md) 中的「VC-05 重构优化 prompt」

**注意：** 优化后必须重新测试所有功能，确保没有引入新问题。

---

## 上下文管理技巧

当一个对话窗口越来越长，AI 的回答质量会下降。以下情况建议开新对话：

- 对话已经超过20轮
- AI 开始"忘记"你之前说的内容
- AI 的建议开始与之前矛盾

**开新对话时，给 AI 的交接信息模板：**

```
我正在做一个[项目名称]，技术栈是[技术栈]。
项目目录结构如下：
[粘贴目录结构]

目前已完成的功能：
- [功能1]
- [功能2]

现在我需要实现/解决：
[当前任务描述]
```

## 避坑指南

| 坑 | 症状 | 解决方法 |
|----|------|---------|
| 代码越改越乱 | 修了一个bug，又出现新bug | 回滚到上一个git提交，重新描述问题 |
| AI不理解你的需求 | 给出的代码完全不对 | 拆分需求，一次只说一个小功能 |
| 上下文太长 | AI开始前后矛盾 | 开新对话，用交接信息模板重新开始 |
| 代码看不懂 | 不知道某行代码做什么 | 用VC-04 prompt逐行要求AI解释 |
| 依赖冲突 | 安装包时报错 | 把完整错误信息（含版本号）给AI |
